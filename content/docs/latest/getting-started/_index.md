## Concepts, Configuration, and Provisioning

Flatcar is configured at _provisioning time_. Two primary configuration formats are used for most use cases (for legacy Ignition 2.x workflows, see the deprecated [Container Linux Config Transpiler](../fb-provision/cl-config/)):

- [Butane](../fb-provision/butane/) is human-readable YAML that must be converted (transpiled) into Ignition config before Flatcar can use it.
- [Ignition](../fb-provision/ignition/boot-process) is machine-readable JSON consumed by Flatcar's first-boot provisioning service. Ignition config is provided as "user data" or "custom data" on cloud platforms, and can also be supplied to private cloud and bare metal installs. It is rarely written by hand; it is usually generated by automation or transpiled from Butane.

You can use Butane to customize your Flatcar deployment to:

- Add custom users and groups.
- Configure storage devices, file systems, swap, and custom files.
- Customize automatic updates and reboot windows.
- Define custom [network configurations](../os-config/network/network-config-with-networkd) with `networkd` and custom [system services](../os-config/systemd/) with `systemd` units.
- Run custom operations or start services at boot.

For a comprehensive discussion of available options, see the [Butane configuration specification](../fb-provision/butane/configuration).

To convert Butane into machine-readable Ignition config, download the latest release from [CoreOS Butane Releases](https://github.com/coreos/butane/releases). Alternatively, use the up-to-date [container image](https://quay.io/repository/coreos/butane), for example: `docker run --rm -i quay.io/coreos/butane:latest < config.yaml > ignition.json`.

After converting Butane to Ignition, provide the config during provisioning through "user data", "custom data", or an equivalent mechanism.

### Automatic updates

Flatcar has automatic updates enabled by default. Instances download and stage new OS versions in the background and can reboot into the updated OS when an update becomes available.
To change this default behavior, including defining reboot windows or disabling reboots, see [update strategies](../updates-releases/releases/update-strategies).
To disable downloading updates altogether, either disable the `update-engine` service via a user-supplied `systemd` config or use an invalid URL in the `SERVER` field of [update.conf](../updates-releases/releases/update-conf).
